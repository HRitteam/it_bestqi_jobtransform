# 平台级大模型路由管理模块拆分计划

为了保证代码质量和开发效率，基于已确认的技术方案，将“平台级大模型路由管理”模块拆分为 3 个逻辑独立的子域。这 3 个子域按照依赖关系从底向上依次为：**模型配置与管理子域**、**核心路由与降级子域**、**日志与计费观测子域**。

## 子域 1：模型配置与管理子域 (Model Configuration & Management)

该子域负责大模型基础信息的增删改查操作、API Key 的安全存储（加密与解密）、模型状态控制以及前端管理界面的搭建。这是整个模块的基础数据底座。

在数据库层面，该子域主要涉及 `llm_models` 表，用于存储模型编码、名称、供应商、API 地址、加密 API Key、优先级、单价以及软删除标记等配置信息。

在后端接口层面，需要在 tRPC 的 `adminLlm` 路由中实现以下功能：获取模型列表（需对 API Key 进行脱敏处理）、新增模型（使用 AES-256-GCM 加密 API Key）、修改模型（支持部分更新，API Key 变更时重新加密）、启用或停用模型，以及软删除模型（需预留检查日志的钩子）。

在前端页面层面，该子域对应 `LlmManager.tsx` 中的“模型管理”标签页。主要组件包括用于展示数据的模型列表表格、用于新增和编辑的弹窗（包含密码框脱敏处理），以及用于切换状态的开关组件。

## 子域 2：核心路由与降级子域 (Core Routing & Fallback)

该子域负责拦截业务层的大模型调用请求，根据子域 1 提供的模型配置，实现基于优先级的动态路由、错误捕获与自动降级切换。这是整个模块的核心业务逻辑。

该子域不涉及新增数据库表，但需要频繁读取 `llm_models` 表的数据。

在核心逻辑模块（即重构后的 `server/_core/llm.ts`）中，需要实现带有短 TTL 的内存缓存机制，以避免频繁查询数据库。同时，需要开发路由调度器来替代原有的硬编码调用方式，负责按优先级遍历模型池。此外，还需要实现错误分类器，以区分触发降级的错误（如 5xx、429、超时、401/403 Key异常）和不触发降级的错误（如 400 参数错误、内容安全拦截）。最后，需要预留异步记录日志的接口，以便对接子域 3。

该子域没有直接对应的前端页面，但需要配合后端进行全链路测试，例如在现有业务功能中触发降级并观察系统表现。

## 子域 3：日志与计费观测子域 (Logging & Billing Observability)

该子域负责记录每次模型调用的详细轨迹（包括成功、失败、切换过程）、Token 消耗和费用估算，并提供总后台的搜索与统计能力。

在数据库层面，该子域主要涉及 `llm_call_logs` 表，用于记录请求ID、企业/用户ID、功能、最终模型、状态、失败原因、切换轨迹、Token 消耗、预估费用以及耗时等信息。

在后端接口层面，需要在 tRPC 的 `adminLlm` 路由中实现供子域 2 异步调用的内部日志记录方法。此外，还需要提供高级搜索接口（支持按企业、手机号、功能、模型、状态、时间、费用等多种条件过滤）和统计接口（计算 Token 消耗总量、预估总费用、调用成功率等）。同时，需要补充子域 1 的逻辑，在删除模型接口中增加对日志的检查，若存在关联日志则拒绝物理删除。

在前端页面层面，该子域对应 `LlmManager.tsx` 中的“调用日志”标签页。主要组件包括多条件组合的高级搜索表单、日志列表数据表格、展示完整请求上下文和 JSON 格式化切换轨迹的日志详情抽屉，以及展示 Token 总量和总费用等信息的顶部统计卡片。

## 开发顺序建议

| 阶段 | 子域 | 核心任务 |
| :--- | :--- | :--- |
| Phase 1 | 模型配置与管理子域 | 建立数据底座，跑通加密解密逻辑，完成前端管理页面开发。 |
| Phase 2 | 核心路由与降级子域 | 重构底层 LLM 调用，接入模型配置数据，实现动态路由和降级逻辑。 |
| Phase 3 | 日志与计费观测子域 | 完善可观测性，记录调用轨迹，完成前端日志查询与统计页面开发。 |
